# MR-006 Plan-Review — slice-034 (Commit-Traceability-Gate, Modul `commits`)

**Datum:** 2026-07-05 · **Reviewer:** unabhängiger Agent (≠ Autor), read-only ·
**Gegenstand:** `docs/plan/planning/done-archive/slice-034-commits-traceability.md`

**Verdikt: 1 HIGH → Start blockiert.** Handwerklich stark (evidence-first sauber belegt,
Muster-treu zu slice-030/033), aber eine Lücke, die dem erklärten Ziel des Slice
widerspricht. HIGH vor Start behoben; MED/LOW eingearbeitet.

## HIGH

- **H1 — Zweite Fundstelle der §4-Regel bleibt undokumentiert → der Slice erzeugt genau den
  „stillen Drift", den er tilgen soll.** Die Traceability-Regel steht an **zwei** normativen
  Stellen: `AGENTS.md` §4 („LH-FA-*, LH-QA-*, ADR-*") **und** `harness/README.md`
  §Traceability rules („PRs/Commits müssen mindestens eine LH-*- oder ADR-*-ID nennen").
  Der Plan zog nur §4 + `harness/README.md` §Sensors nach — die §Traceability rules blieben
  unberührt. Ergebnis: drei Formulierungen mit zwei Kennungs-Sets (amendiertes §4 =
  LH/ADR/slice/MR, README-§Traceability = nur LH/ADR, Gate-id-patterns = slice/ADR/MR/LH) —
  exakt der Drift, dessen Vermeidung der Plan als Erfolgskriterium nennt. **Fix:** README
  §Traceability rules in die Amendment-Fläche aufnehmen (Kennungs-Set angleichen +
  Gate-Bindung nennen), symmetrisch zu §4.

## MED

- **M1 — §2.6-Argument deckt die Verbreiterung des Kennungs-Sets nicht ab.** Der ADR-Verzicht
  wird mit „kein ADR für ein strengeres Gate" begründet; das Gate ist gegenüber dem alten
  Wortlaut aber zugleich **permissiver** (`slice-\d{3}` allein genügt). Die transitive
  Traceability (commit→slice→LH/ADR) trägt nicht durchgängig — Steering-Slices (wie slice-034
  selbst, `lastenheft_refs`/`adr_refs` leer) sind auf keine LH/ADR-ID rückführbar. Verteidigbar
  (Null-Enforcement → Enforcement = netto strenger; MR-013/MR-014-Präzedenz), aber **MR-015
  soll explizit** festhalten: Steering-/Prozess-Slices sind legitim anforderungsfrei, `slice-*`
  ist ihr Traceability-Anker.
- **M2 — Keine Roadmap-Registrierung in der DoD.** Jedes bisherige harness-steering-Quergewerk
  trägt eine Zeile in `roadmap.md` (018a/b/c, 028/029, 030, 027, 031, 033). **Fix:** DoD-Zeile
  „Roadmap-Quergewerk-Eintrag (Muster slice-033-Zeile)".

## LOW / INFO

- **L1 — `slice-\d+` vs. `slice-\d{3}`:** die commits-`id-patterns` nutzten `slice-\d+`, während
  das `matrix`-Modul derselben `.d-check.yml` `token: 'slice-\d{3}'` nutzt und MR-002 3-stellige
  Slice-IDs deklariert. Funktional harmlos, aber angleichen auf `slice-\d{3}` vermeidet
  Cross-Config-Divergenz. **Behoben.**
- **L2 — „gates unberührt" per Docker-Lauf belegen:** der `commits:`-Block ist bereits in
  `.d-check.yml`, aber **nicht** im aktiven `modules:`-Set → `make docs-check` ignoriert ihn.
  Autor soll `make docs-check` grün mit dem Block zeigen (Risiko niedrig).
- **INFO — Reihenfolge:** Teil der Impl (der `.d-check.yml`-Block) liegt bereits im Arbeitsbaum
  vor MR-006-Freigabe (für die Messung vertretbar); jede MR-006-getriebene `id-patterns`-Änderung
  muss in die schon modifizierte Datei zurückfließen.
