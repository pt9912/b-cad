## MED

- **M1 — Kopplung an die §2.8-Zwei-Commit-Regel un-analysiert; jede Ruhe⇄Aktiv-Transition erzeugt
  einen committeten `planning-drift`-Snapshot (slice-035-M1-Klasse, hier NEU eingeführt).** Der
  `git mv slice-NNN in-progress→done` des **letzten** Slice ist ein reiner Move-Commit; das
  Marker-Setzen ist eine `roadmap.md`-Content-Edit → separater Nachzug-Commit. Dazwischen: S1
  (`in-progress` leer + Marker abwesend) → `planning-drift` (empirisch). Symmetrisch beim Öffnen
  des ersten Slice (S3: Slice + Marker → Drift bis Nachzug). Heute latent (`gates` läuft auf dem
  Arbeitsbaum-Endzustand), aber mit der angekündigten ersten `.github/workflows`-Datei (per-Commit)
  feuert das Gate auf dem Zwischen-Commit. **Fix:** §6/MR-017 (a) benennt den transienten Zustand,
  (b) schreibt die Commit-Sequenz vor. **Auflösung (Reviewer):** der Marker-Toggle darf **im
  selben Commit wie der `git mv` reiten** — Rename-Detection ist **per-Datei**, eine `roadmap.md`-
  Edit senkt die Rename-Similarität von `slice-NNN` **nicht** → §2.8 (saubere Move-Erkennung) wird
  nicht verletzt, und es entsteht **kein** transienter Drift-Snapshot. Zusätzlich: per-Commit-CI
  wertet `planning` sinnvoll auf dem PR-finalen HEAD.
- **M2 — Toggle-Pflicht nicht an den kanonischen Lifecycle-Stellen (Auffindbarkeit einer
  gate-blockierenden Pflicht).** Der Plan verankert sie in `planning/README.md` + Roadmap-Kommentar

  Heimat der Lifecycle-Bewegung). Wer den letzten Slice schließt, operiert aus §2.8 und wird nicht
  an den Toggle erinnert → blind in den roten `planning-drift`. Bei einem **Gate-Member** verschärft.
  **Fix:** Ein-Zeilen-Referenz in AGENTS §2.8 auf die Ruhe-Marker-Toggle-Pflicht; in die DoD.
