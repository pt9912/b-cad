## Negativbefunde (geprüft, ohne Befund)

- **Kein Schema-Wechsel (Claim 1):** `materials`-Tabelle + `material_id` (`ON DELETE RESTRICT`) an walls/roofs/slabs sind in `schema.sql` **und** `data-model.yaml` vorhanden; `make schema-check` difft `schema.sql` gegen `d-migrate(data-model.yaml)` (Generator-Pin 0.9.7). **Kein Edit nötig → schema-check bleibt grün.** Korrekt.
- **Spaltenliste (Claim 2):** `materials` hat genau 10 Spalten; die INSERT/SELECT-Liste (8 + `project_id`-Literal, `density` ausgelassen) ist konsistent. `density` ist echte Schema-Spalte ohne Feld in `model::Material`.
- **NULL-vs-0.0 (Claim 3):** **Null** Vorkommen von `sqlite3_column_type`/`SQLITE_NULL`/`bind_null` im Repository — alle Reads nutzen `sqlite3_column_double` direkt. Das Risiko ist **real und heute ungedeckt**; der `SQLITE_NULL`-Check ist der korrekte Fix; AK (b) ist ein gültiger Sensor.
- **FK-Insert-Reihenfolge (Claim 4):** `PRAGMA foreign_keys=ON` vor `BEGIN`; aktuelle Reihenfolge project→storeys→walls→… ohne materials-Schritt. `insertMaterials` **muss** vor walls/roofs/slabs. Korrekt.
- **ID-Erhalt (Claim 5):** bestehende Inserts binden `static_cast<int>(id)`; `MaterialId` ist `enum class : int`. ID-Erhalt sound.
- **Kein neuer Parse-Pfad (Claim 7):** alle `materials`-Spalten REAL/TEXT, kein JSON → keine `std::stod`-Suffix-/Enum-Mapper-Exposition; Totalitäts-Fokus = NULL-Behandlung. Korrekt.
- **Lastenheft unberührt / `MR-010` (Claim 8):** `LH-FA-MAT-*` in `data-model.yaml` ist Beschreibungstext, keine AK; MAT/BLD schon auf Niveau → kein Header-Bump. Spec-Edit auf spezifikation.md §1 + Historie begrenzt. Korrekt.
- **Test-Passung (Claim 9):** `sampleBuilding()` + White-Box `corruptColumn()` existieren; Round-Trip + `EXPECT_THROW`-Muster tragen die Material-AK. `Building::materials` + `optional<MaterialId>` an allen drei Bauteilen vorhanden.
- **Scope/Sizing (modul-05):** ~2 Code-Schichten (Adapter + Tests) + Spec/Gates-Housekeeping; `wall_types`/`stairs`/`openings` korrekt ausgeschlossen. Eine Review-Sitzung plausibel. Die höhere Code-Review-Latte ist durch die Persistenz-Fehlerklasse gerechtfertigt.
